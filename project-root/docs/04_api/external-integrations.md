# 外部服务与能力集成说明

## 1. 文档用途

本文件用于统一说明本项目涉及的外部系统、第三方 API、模型服务及其接入原则，作为系统架构文档中“外部集成”部分的补充说明。

本文件主要回答以下问题：

1. 当前项目可能需要接入哪些外部能力
2. 这些外部能力分别服务于哪些业务功能
3. 当前阶段哪些能力已经确定、哪些尚未确定
4. 在外部 API 尚未最终选型前，系统如何预留扩展空间
5. 接入第三方服务时需要遵循哪些设计、安全与协作规则

本文件不替代详细接口文档，也不直接记录具体厂商 API 的全部技术细节。  
如后续外部服务正式确定，可在本文件基础上继续补充更细的接入说明。

---

## 2. 当前状态说明

截至当前阶段，项目已经完成核心需求、模块划分、数据设计和 API 设计的基础文档整理，系统已具备进入 MVP 开发阶段的条件。

但部分创新功能和增强能力依赖外部服务，当前仍处于“预留设计、暂未最终选型”状态，主要包括：

1. **地图类服务**
   - 用于增强路线展示、地理位置表达、轨迹记录、地图可视化等能力
   - 当前尚未最终确定具体接入厂商

2. **AI / 多模态模型服务**
   - 用于日记草稿生成、图片摘要、多人协同规划总结、推荐理由解释等能力
   - 当前尚未最终确定具体模型提供方或调用方式

### 当前总体原则

- **MVP 主链路不依赖外部 API 才能跑通**
- 外部 API 当前作为**增强能力**或**创新功能支撑能力**
- 在选型未最终确定前，系统先做**接口预留、模块解耦、配置预留、Mock 支持**
- 待后续选型明确后，再补充具体厂商接入实现

## 4.2 AI 类能力与业务模块的关系

- `diary-module.md`
- `recommend-module.md`

### 主要支撑场景

根据用户上传图片、路线记录、文字提示等内容，生成旅游日记草稿。

根据用户上传的旅游图片，提取图片内容摘要和标签。

#### 场景 C：路线回顾总结
根据历史路线记录生成路线回顾或总结文案。

对应预留接口：
- `POST /api/v1/ai/route-review`

#### 场景 D：推荐理由解释
对推荐结果提供可读性更强的解释信息。

对应预留接口：
- `POST /api/v1/ai/recommend-reason`

#### 场景 E：多人规划协同总结
对多人规划的讨论结果、偏好冲突和最终建议进行总结输出。

对应预留接口：
- `POST /api/v1/ai/group-plan-summary`

## 5. 当前建议纳入管理的外部集成项

本节用于记录“项目层面已经识别到、后续可能接入”的外部服务项。

## 5.1 地图服务（高德地图 AMap）

### 集成名称
高德地图 Web 服务 API

### 当前状态
**已确定选型：高德地图 API**

### 候选厂商对比
- **高德地图 API**（已选定）
  - 静态地图 API：地图底图展示
  - 搜索 POI 2.0：地点查询、附近设施查找
  - 路径规划 2.0：单目标/多目标路径规划
  - 地理/逆地理编码：地址转坐标，数据初始化
  - 输入提示：搜索联想
  - JavaScript API：前端交互地图组件
- 百度地图开放平台（备选）

### 主要用途
#### P0 基础功能支撑
- **地理编码/逆地理编码**：用于数据初始化阶段将景点、学校、设施的地址转换为经纬度坐标
- **搜索 POI 2.0**：支持用户输入地点名称、类别、关键字进行查询（F-MAP-015）
- **路径规划 2.0**：作为内部路径规划算法的验证对照，或作为降级方案

#### P1 增强功能支撑
- **静态地图**：地图底图展示，为路径规划结果提供可视化背景
- **输入提示**：搜索联想功能，提升用户查询体验
- **类别过滤**：结合 POI 搜索实现设施类别筛选（F-MAP-018）

#### P2 创新功能支撑
- **JavaScript API**：前端交互式地图，支持点击选点、轨迹展示、路线高亮等
- **轨迹记录**：配合定位能力记录用户实际移动轨迹
- **室内导航**：楼层间导航（需评估高德室内地图覆盖范围）

### 核心 API 列表
| API 名称 | 用途 | 优先级 | 对应功能编号 |
|---------|------|--------|-------------|
| 静态地图 | 地图底图展示 | P1 | F-MAP-003, F-MAP-010 |
| 搜索 POI 2.0 | 地点查询、附近设施查找 | P0 | F-MAP-015, F-MAP-016, F-MAP-018 |
| 路径规划 2.0 | 单目标/多目标路径规划 | P0/P1 | F-MAP-001, F-MAP-008 |
| 地理/逆地理编码 | 地址转坐标，数据初始化 | P0 | 数据初始化 |
| 输入提示 | 搜索联想 | P1 | F-MAP-015 |
| JavaScript API | 前端交互地图 | P1 | F-MAP-003, F-MAP-010 |

### API 文档链接
- [静态地图 API](https://lbs.amap.com/api/webservice/guide/api/staticmaps)
- [搜索 POI 2.0](https://lbs.amap.com/api/webservice/guide/api-advanced/newpoisearch)
- [路径规划 2.0](https://lbs.amap.com/api/webservice/guide/api/newroute)
- [地理/逆地理编码](https://lbs.amap.com/api/webservice/guide/api/georegeo)
- [输入提示](https://lbs.amap.com/api/webservice/guide/api-advanced/inputtips)
- [错误码说明](https://lbs.amap.com/api/webservice/guide/tools/info)
- [JavaScript API](https://lbs.amap.com/api/javascript-api-v2/summary)

### 当前是否阻塞 MVP
**部分阻塞**：
- P0 阶段的地点查询（F-MAP-015）和地图选点查询（F-MAP-016）可使用高德 POI 搜索 API 快速实现
- 但核心路径规划算法（F-MAP-001, F-MAP-002）仍需基于内部图结构自主实现，以满足课程要求
- 地图底图展示可作为 P1 增强项，不影响 MVP 主线跑通

### 当前处理策略
1. **核心算法自主实现**：路径规划核心算法（Dijkstra、A*等）必须基于自建设计的数据结构实现，符合课程要求
2. **POI 数据可借助外部 API**：地点查询、附近设施查找等功能可直接调用高德 POI 搜索 API，减少数据维护成本
3. **坐标数据初始化**：使用高德地理编码 API 批量转换景点、学校、设施的地址为经纬度坐标
4. **前后端分离设计**：
   - 后端封装 `MapService` 接口，隐藏具体 API 调用细节
   - 前端通过项目后端 API 间接调用高德能力，不直接暴露 API Key
5. **配置化管理**：
   - `MAP_PROVIDER=amap`
   - `AMAP_API_KEY`（通过环境变量注入，不得提交到 Git）
   - `AMAP_API_BASE_URL=https://restapi.amap.com/v3`
6. **降级方案**：
   - 若高德 API 不可用或额度不足，地点查询可降级为本地数据库模糊匹配
   - 路径规划始终使用内部算法，不依赖外部 API

### 注意事项
1. **API Key 安全**：密钥必须通过环境变量或本地配置文件注入，严禁提交到 Git 仓库
2. **调用频率限制**：需注意高德 API 的 QPS 限制和每日配额，必要时实现缓存和限流
3. **数据合规性**：POI 数据缓存需遵守高德地图 API 使用条款
4. **课程要求对齐**：核心路径规划算法必须自主实现，外部 API 仅作为数据源和增强能力

## 6. 外部集成设计原则

为避免后续外部服务接入造成系统结构混乱，项目约定所有外部集成都应遵循以下原则。

## 6.1 解耦原则

业务模块不得直接强耦合具体第三方厂商 SDK 或调用细节。  
应通过统一的服务抽象层进行封装，例如：

- `MapService`
- `AIService`

业务层只依赖抽象接口，不直接依赖某一具体平台。

---

## 6.2 可替换原则

选型未定或未来可能变更的外部服务，应采用“可替换”设计：

- 不把厂商名称写死到核心业务逻辑中
- 不在多个业务模块中散落第三方调用代码
- 第三方能力变更时，尽量只改适配层，不改业务主流程

---

## 6.3 配置化原则

所有外部服务配置都应放在配置文件或环境变量中管理，不允许硬编码在代码中。  
至少应包含以下配置项：

### 地图服务配置
- `MAP_PROVIDER`
- `MAP_API_KEY`
- `MAP_API_BASE_URL`

### AI 服务配置
- `AI_PROVIDER`
- `AI_API_KEY`
- `AI_API_BASE_URL`
- `AI_MODEL_NAME`

---

## 6.4 降级原则

如果外部服务暂不可用、未配置、调用失败或额度不足，系统应具备基本降级能力。

### 地图服务降级
- 若地图服务不可用，仍可使用系统内部图结构完成路径规划
- 前端可先使用文本 / 列表方式展示路线结果
- 暂不依赖真实地图底图也能完成基本功能验证

### AI 服务降级
- 若 AI 服务不可用，可返回固定模板结果或 mock 数据
- 日记模块仍应支持用户手动编写和发布
- 推荐模块不能因为 AI 解释不可用而阻塞主推荐结果返回

---

## 6.5 安全原则

所有外部服务接入都必须符合项目安全规范：

1. API Key、Secret 等敏感信息不得提交到 Git 仓库
2. 密钥必须通过本地环境变量或安全配置文件注入
3. 外部响应数据不得直接信任，必要时需做校验和过滤
4. 调用外部 API 时应考虑超时、重试、限流和异常处理
5. 涉及用户图片、文本等数据上传时，应注意隐私和最小必要原则

具体规则参见：
- `project-root/security-rules.md`

---

## 6.6 Mock 优先原则

在外部服务尚未确定前，允许先基于 mock / stub 完成开发与联调。  
Mock 主要用于以下阶段：

- 前后端联调
- 功能链路演示
- 接口结构验证
- 中期汇报 / 课程演示准备

当前建议：

- AI 接口先返回示例生成结果
- 地图能力先使用内部数据结构或静态示意数据

## 7. 接入落点与代码组织建议

## 7.1 后端建议

后端建议在服务层单独设置外部集成适配层，例如：

- `integration/map/`
- `integration/ai/`

或按服务命名为：

- `MapService`
- `MapServiceImpl`
- `AIService`
- `AIServiceImpl`

### 建议职责划分
- Controller：接收请求，返回统一响应
- Business Service：处理业务逻辑
- Integration Service：调用第三方 API / 模型服务
- Mapper / Repository：处理本地数据库访问

---

## 7.2 前端建议

前端不直接感知具体第三方厂商。  
前端只应调用本项目后端 API，例如：

- `/api/v1/routes/...`
- `/api/v1/ai/...`

如果后续需要接入第三方地图组件，建议也尽量通过统一页面组件进行封装，避免地图厂商逻辑散落到多个页面中。
